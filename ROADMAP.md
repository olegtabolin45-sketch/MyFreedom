# 🚀 AETERNA ANALYTICS — Roadmap

> Аналог Snowball Income, но лучше. Инструмент для анализа капитала и построения пассивного дохода.
> Принципы: **безопасность с первого дня**, **горизонтальное масштабирование**, **чистый Apple-style интерфейс**.

---

## 📊 Текущее состояние (что уже есть)

- ✅ Фронтенд: лендинг + формы регистрации/входа (`index.html`), калькулятор трека (`dashboard.html`)
- ✅ Бэкенд: FastAPI с эндпоинтами `register / login / user/status / onboarding` (`server.py`)
- ✅ JWT-авторизация + bcrypt-хэширование паролей
- ✅ Хранилище: Trino + Iceberg

---

## ⚠️ КРИТИЧНО исправить до любого нового функционала

Это нужно закрыть **до** написания новых фич — иначе будем масштабировать дыры.

- [ ] **SQL-инъекции.** Сейчас запросы собираются через `"... '%s'" % value` с ручным экранированием (`escape_sql`). Это ненадёжно. Перейти на **параметризованные запросы** (`cursor.execute(query, params)`).
- [ ] **Секреты в коде.** `JWT_SECRET` захардкожен в `server.py`. Вынести в переменные окружения (`.env` + `python-dotenv`), добавить `.env` в `.gitignore`.
- [ ] **CORS.** Сейчас `allow_origins=["*"]` вместе с `allow_credentials=True` — небезопасно. Указать конкретные домены.
- [ ] **Trino — не для пользователей.** Trino/Iceberg — аналитический движок, он не предназначен для транзакционных операций (регистрация, логин). Завести **PostgreSQL** для пользователей/сессий, а Trino оставить только для аналитики капитала.
- [ ] **Нет rate limiting.** Добавить ограничение запросов (защита от брутфорса логина).
- [ ] **Логи с чувствительными данными.** Убрать вывод ошибок с деталями (`[SECURITY ALERT] ... {e}`) в продакшене.

## 🏗️ Этап 1 — Фундамент архитектуры

- [ ] Разделить монолит `server.py` на модули: `routers/`, `models/`, `schemas/`, `services/`, `db/`, `core/`
- [ ] Подключить PostgreSQL + миграции (Alembic)
- [ ] Файл зависимостей `requirements.txt` (зафиксировать версии: `bcrypt==4.0.1` и т.д.)
- [ ] Конфигурация через `pydantic-settings` + `.env`
- [ ] Docker + `docker-compose.yml` (app + postgres + trino + redis)
- [ ] Структурированное логирование (`structlog` / `loguru`)

## 🔐 Этап 2 — Безопасность (углублённо)

- [ ] Refresh-токены + короткоживущие access-токены (сейчас один токен на 60 мин)
- [ ] Хранение refresh-токенов и blacklist в **Redis**
- [ ] Подтверждение email при регистрации
- [ ] Сброс пароля по email
- [ ] Двухфакторная аутентификация (2FA / TOTP)
- [ ] Заголовки безопасности (HSTS, CSP, X-Frame-Options) через middleware
- [ ] Защита от CSRF
- [ ] Аудит-лог действий пользователя
- [ ] Валидация и санитизация **всех** входных данных

## 🛡️ Этап 3 — Защита от DDoS и нагрузки

> Важно: полноценная защита от DDoS строится на уровне инфраструктуры, а не только кода.

- [ ] **CDN + WAF** (Cloudflare / подобное) — первая линия защиты от DDoS, фильтрация трафика
- [ ] Reverse-proxy (Nginx) с rate limiting и таймаутами
- [ ] Кэширование тяжёлых аналитических запросов в Redis
- [ ] Connection pooling к БД
- [ ] Защита от ботов (капча на регистрации/логине)
- [ ] Мониторинг аномального трафика

## 📈 Этап 4 — Основной функционал (аналитика капитала)

- [ ] Интеграция с источниками котировок (API бирж)
